# Check-2 gaps found in #117 (KIRIIYARASAN) — four fixes

## Context

A live review of applicant **#117** surfaced four defects. Three are systemic (they bite a
measurable slice of the cohort); one is a latent fragility in a parser. None is about #117 alone.

What the review found, in the order it matters:

1. **The water-bill parser can never read an address, so it re-asks forever.**
   `doc_parse._parse_water` returns a hardcoded `'address': ''` (doc_parse.py:503), but
   `income_engine._bill_needs_upload` re-asks for any bill whose address is unreadable
   (income_engine.py:2507). A clean Air Selangor PDF is therefore permanently "address unreadable".
   #117 re-sent the same PDF **three times** and only escaped by *photographing the PDF* (the photo
   routes to Gemini, which read the address fine). **6 live water bills** are read this way and all
   6 have a blank address — apps **30, 36, 66, 80, 82, 95**. #36 has an open, unsatisfiable request
   right now; #66 had to be manually waived. The electricity parser (doc_parse.py:245-263) reads its
   address block properly — water was simply written without one.

2. **The offer letter's pathway is never really compared with the declaration.**
   #117's offer summons him to **KOLEJ TINGKATAN ENAM GOMBAK, Bidang SAINS**; he declared **SMK (P)
   TEMENGGONG IBRAHIM, sains_sosial**. Neither difference was caught, for two independent reasons:
   - The **stream/track is not in the comparison at all** — `offer_pathway_match` weighs only
     programme and institution. Two other students carry the same unflagged clash: **#33 (awarded)**
     and **#99**, both `sains_sosial` against an offer that says SAINS.
   - **The comparison is circular.** `autofill_pathway_from_offer` writes `chosen_programme` from
     the offer, and `_declared_pathway` then *prefers* `chosen_programme` as "the declaration".
     **45 applications** carry `source: offer_letter_auto`. Worse, when the offer's institution
     fails to extract, autofill falls back to the student's *own* declared school and still stamps
     it `offer_letter_auto` — so #117's cockpit shows the SMK as if the offer said so.

3. **A pensioner is invisible to the means test.** `retired` sits in `NON_EARNING` (family.py:86),
   so `member_income_status` returns "satisfied": we never ask what he draws and never count it.
   There is no pension question among the 19 clarify codes. **12 applications** have a retired /
   unable-to-work parent. The error runs toward *understating* income, which is the direction a
   fiscal steward should mind.

4. **The household under-count query is suppressed at exactly the case that matters.**
   `_ROSTER_UNDERCOUNT_MARGIN = 2` (income_engine.py:2694). #117 describes 5 people against a stated
   6, so the gap is 1 and nothing was asked. The household count is the per-capita denominator.

**Owner decisions taken (2026-07-14):**
- Pension: **ask, then ask for proof** — mirror the #126 ask-first→doc pattern, not a bare clarify.
- Roster margin: **lower to 1**, accepting ~11 more students asked.
- Existing cases: fix #36, re-run the 6 water bills, surface #33/#99 to officers, re-run #117.

### Two side-findings, worth folding in cheaply
- **The sister's EPF is never requested.** `employed_epf_members` (income_engine.py:2457) loops only
  `('father','mother')` while the *payslip* request was generalised to the whole household in Phase
  2C. `sister_epf_missing` / `brother_epf_missing` / `guardian_epf_missing` exist in
  `_MEMBER_EPF_CODE` and are unreachable.
- **#117's STR docs read "name not found"** — a stale false flag. `vision.reference_names` (which
  adds roster parents) landed 14 Jul; he uploaded on the 12th. A cockpit Re-run clears it.

## Fix 1 — Water bill: read the address, or bail to Gemini

**File:** `halatuju_api/apps/scholarship/doc_parse.py` (`_parse_water`, ~line 495).

Add an address block read, following the shape of `_parse_electricity` (doc_parse.py:245-263, which
anchors on `ALAMAT POS` and takes the block to `TARIKH BIL`). Air Selangor prints the holder name and
supply address as a block near the account number.

**The load-bearing rule:** if the address cannot be read, **`return None`** so the doc falls through
to Gemini — the module's documented conservative-bail convention ("None → Gemini"). Never again
return a dict with a blank address. Gemini already reads these correctly: **46 of the 'ai'-captured
water bills have an address**, and it read #117's photo of the very same bill.

Do **not** loosen `_bill_needs_upload` — "no address" genuinely is an unusable bill; the bug is that
we were manufacturing the blank ourselves.

**Validation before shipping (important — this is a heuristic on real OCR):** use the existing
`eval/capture_ocr.py` pipeline against the 6 known docs (ids **1248, 928, 938, 850, 1469, 1007**)
plus #117's superseded ones (**2035, 2039, 2042**), and confirm each yields an address and that the
`amount` / `bill_date` reads are unchanged. Precedent: the S15 address heuristic took three passes
because synthetic fixtures could not reproduce real OCR — validate on real text, not fixtures.

**Data follow-up (post-deploy, on the LIVE service only):** cockpit Re-run the 6 bills. #36's open
`water_bill_recheck` auto-resolves when the gap clears (`sync_check2_queries` closes a doc item whose
gap has gone).

## Fix 2 — Pathway: read the offer properly, compare the track, break the circularity

Four sub-parts. (a) and (b) are #117-shaped; (c) and (d) are the systemic half.

### (a) Harden the STPM offer parser
**File:** `halatuju_api/apps/scholarship/offer_parse.py` (`_parse_stpm`, lines 75-100).

The parser is **line-anchored**, and this PDF's OCR text layer joined each label to its value on one
line. That single quirk broke three fields:
- **Name** — `_value_after_label` captured `(.*)` to end-of-line, giving
  `"KIRIIYARASAN A/L MUNIANDY No. Kad Pengenalan: 080111140499"`. Truncate the captured value at a
  following label (split on `No.?\s*Kad\s*Pengenalan`).
- **Institution** — the regex requires a line *starting* with `SEKOLAH|SMK|KOLEJ|MAKTAB`; here the
  value sits after the `Pusat Tingkatan Enam` label on the same line. Prefer a value-after-label read
  of `Pusat Tingkatan Enam`, falling back to today's line-start scan.
- **Reporting date** — the loop skips any line containing `TARIKH` (intending to skip the print-date
  header), which also kills the real `Tarikh Lapor Diri` line. Read the value after
  `Tarikh Lapor Diri` explicitly; keep the `CETAK` / offer-date exclusions as the fallback.

Also **capture the Bidang** (`2.1 Bidang  SAINS`) into a `stream` field — the deterministic parser
never emits one today. The Gemini schema already has `stream` (vision.py:1209) and **20 of 22** live
STPM offers carry it, so this is the deterministic path catching up with the AI path.

Same principle throughout: read label→value; do not depend on line boundaries.

### (b) Put the track into the comparison
**File:** `halatuju_api/apps/scholarship/pathway_engine.py`.

- `student_offer_check` (line 294) — surface `stream` alongside `programme` / `institution`.
- `offer_pathway_match` (line 83) — add a **third dimension**: the offer's `stream` vs the
  application's `pre_u_track`. Normalise both (`"SAINS SOSIAL"` → `sains_sosial`). A clash on the
  track is a `mismatch`, same as a clash on institution or programme.

The existing chain then works unchanged: `verdict_engine` emits `pathway_confirm` on a mismatch, and
`check2_queries._sync_pathway_confirm` (line 420) mirrors it into the student queue as the one-tap
"is this your final course?" confirm. **No new reason code is needed** — that is the point of routing
through the existing clash detector.

### (c) Break the circularity
**File:** `halatuju_api/apps/scholarship/pathway_engine.py` (`_declared_pathway`, line 101).

When `chosen_programme.source == 'offer_letter_auto'`, that value **is the offer** — it is not a
declaration and must not be compared against itself. In that case fall back to the student's own
`pre_u_institution` / `pre_u_track`. Affects **45 applications**, none adversely: it can only reveal
clashes we are currently blind to.

### (d) Stop autofill from laundering the declaration
**File:** `halatuju_api/apps/scholarship/services.py` (`autofill_pathway_from_offer`, line 1102;
`_canonical_preu_institution`, ~line 1172).

For STPM, `clean_school_name(inst, application.pre_u_institution)` falls back to the student's
declared school when the offer's institution is blank — and the result is still stamped
`source: 'offer_letter_auto'`. Do not write an institution into `chosen_programme` that did not come
from the offer; leave it blank rather than attribute the student's own answer to the letter. With (a)
landed, #117's offer yields a real institution anyway, but the laundering must not survive.

**Officer surfacing for #33 / #99 (owner-approved):** both are past the Completed stage, so the stage
gate correctly withholds a *student* query. Their clash should appear as an officer-visible pathway
note on the cockpit (the verdict's `pathway_confirm` evidence line already renders there once (b)
detects it) — verify after the re-run rather than building a new surface.

## Fix 3 — The pensioner: ask, then ask for proof

Mirrors #126 exactly: **ask first, hear the answer, and only then request the document.** Never
demand a payslip from someone who cannot have one.

### The occupation class
**File:** `halatuju_api/apps/scholarship/family.py` (line 86).

Add `BENEFIT_OCC = frozenset({'retired', 'unable'})` beside `NON_EARNING` / `INFORMAL_OCC`. Leave
`NON_EARNING` membership intact so nothing else re-bands; `BENEFIT_OCC` is a new, narrower lens:
*this person is not employed, but may draw a recurring benefit.*

### The clarify
**File:** `halatuju_api/apps/scholarship/income_engine.py`
- `pension_context(application)` → a params dict, or `None` when there is no gap. Model on
  `informal_income_context` (line 2683) — the `ctx is not None` idiom doubles as the gap detector,
  as `high_utility_expense_context` already does at check2_queries.py:203-205. Fires when a member's
  occupation is in `BENEFIT_OCC` **and** no income evidence is on file for them.
- `pension_claim(text)` → `'yes' | 'no' | 'unclear'`. A near-copy of `payslip_claim` — **negation
  checked first** (that ordering is what stops the #130 dead-end from re-forming).
- `pension_claimed(application, member)` — reads `params.pension_claim` off the resolved clarify;
  defensive `getattr(..., None)` (the #126 SimpleNamespace lesson).

**File:** `halatuju_api/apps/scholarship/views.py` — the resolve endpoint already classifies
`informal_income_detail` answers at resolve time and writes `params.payslip_claim`. Add the twin
branch for `pension_amount_unknown` → `params.pension_claim`.

### The doc request (only after a "yes")
**File:** `halatuju_api/apps/scholarship/check2_queries.py`
- `DOC_SPECS` — `father_pension_proof_missing` / `mother_pension_proof_missing`, both
  `{'member': ..., 'doc_type': 'salary_slip'}`. **No new doc type, no migration**: the salary-slip
  extraction prompt already states it accepts "a government benefit / pension statement … which
  counts as household income too" (vision.py:1364). Separate codes (rather than reusing
  `father_income_proof_missing`) purely so the copy can say *pension statement*, not *salary slip*.
- New `_MEMBER_PENSION_CODE` map beside `_MEMBER_EPF_CODE` / `_MEMBER_PROOF_CODE` (line 122).
- `_gap_sets` — `proof_wanted.add(...)` for each `BENEFIT_OCC` member where `pension_claimed()` is
  `'yes'` and no income doc is on file.

### The clarify wiring — the four links
1. `CLARIFY_SPECS['pension_amount_unknown'] = {'fact': 'income'}` (line 33).
2. `_gap_sets` — import the detector (lines 161-174) + `gaps.add(...)`.
3. **`_CLARIFY_ORDER` (line 136)** — insert **high**, next to `informal_income_detail`. `MAX_CLARIFY
   = 3` and `sync_check2_queries` iterates `_CLARIFY_ORDER`, **not** `CLARIFY_SPECS`: a code omitted
   here is **silently never raised**. This is the trap; there is no test guarding it.
4. `_clarify_params` (line 250) — branch returning `pension_context(...)` so the copy names the
   member.

Then: `actionCentre.ts KNOWN_CODES` (line 111) + **9 i18n leaves per code** — 3 locales ×
[`scholarship.actionCentre.item.<code>.title`, `.desc`, `admin.scholarship.verdict.item.<code>`].
Note the asymmetry: the first namespace is an object, the third a flat string. Flat `{member}`
placeholders only — **the custom `t` has no ICU** (`no-icu-messageformat.test.ts` enforces this).
Tamil is a first draft, queued for owner review as usual.

### While here — the unreachable sibling EPF (cheap, same file)
`employed_epf_members` (income_engine.py:2457) — extend the loop from `('father','mother')` to the
full roster, exactly as `household_status_gaps` (line 1999) already does. The per-member codes
(`sister_epf_missing` etc.) already exist in `_MEMBER_EPF_CODE`; they are simply unreachable. The new
slip-driven KWSP rule (`slip_epf_evidence`) gates the ask, so a pensionable civil servant with no
KWSP line is correctly never asked.

## Fix 4 — Roster margin

**File:** `halatuju_api/apps/scholarship/income_engine.py` line 2694.

`_ROSTER_UNDERCOUNT_MARGIN = 2` → `1`, and update the docstring/comment on
`household_roster_undercount` (line 2697), which currently justifies the 2 ("an under-count of one is
common/benign"). That justification is now explicitly overruled by the owner: the household count is
the per-capita denominator, so one unaccounted person changes the means test.

**Blast radius:** 11 additional students reach the gap; 14 already clear the old bar. All are
clarifies, capped at 3 concurrent — so the *priority* of `household_roster_undercount` in
`_CLARIFY_ORDER` (line 142) now matters more. It already sits above the comfort items; leave it.

## A guardrail worth adding while we are in here

**File:** `halatuju_api/apps/scholarship/tests/test_check2_queries.py`

Assert **`set(CLARIFY_SPECS) == set(_CLARIFY_ORDER)`**. A clarify present in `CLARIFY_SPECS` and
`_gap_sets` but missing from `_CLARIFY_ORDER` is raised **never**, silently, with no error anywhere.
This plan adds a code to all three; the next one may not.

(Second, optional: `actionCentre.test.ts:127` reads **`en.json` only**, so a missing `ms`/`ta`
Action-Centre string does **not** fail jest — unlike the officer namespace, which is parity-tested.
Extending it to all three locales would close a real hole.)

## Not in scope
- **Counting the pension in the means test** (`income_per_capita` / headroom). The owner chose
  ask-and-evidence, not re-banding. Once pension statements are on file, feeding them in is a clean,
  separate follow-up — and it is the step that actually closes the understatement.
- **Any migration.** Every change here is code + i18n; the pension proof reuses the `salary_slip`
  slot, which the extraction prompt already treats as accepting a pension statement.
