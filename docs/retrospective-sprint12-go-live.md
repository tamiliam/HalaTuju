## What Was Built / Done

The lawyer-gated go-live of the entire B40 Phase E/F sponsor programme — owner-authorised to ship with the **current
draft consent wording** (lawyer-vetted text + a `CONSENT_VERSION` bump to follow). The 25 held-local commits (Sprints
1–11) were deployed as one batch, in migrate-first order:

1. **Migrations `0049`–`0054` → prod (MCP, migrate-first).** Additive columns (`onboarded_at`, `assigned_at`, sponsor
   notify fields, `realtime_notified_at`) + 6 new tables. Each new-model migration applied the DDL **and enabled RLS in
   the same transaction** (so no table ever existed without RLS — critical, since the public schema is exposed via
   PostgREST and `graduation_messages.raw_text` / `sponsor_referrals.invitee_email` hold PII). Hand-authored Postgres DDL
   matching prod's `bigint GENERATED BY DEFAULT AS IDENTITY` PK convention + `DEFERRABLE INITIALLY DEFERRED` FKs;
   `django_migrations` rows recorded so Django sees them applied (the TD-058 contenttypes workaround — no `manage.py
   migrate` against prod, so no failing `post_migrate`). Verified table-by-table after applying.
2. **`git push` → Cloud Build.** Both api + web triggers built SUCCESS (~6 min); api revision `…00325`.
3. **Flag flip.** `SPONSOR_POOL_ENABLED=true` via `gcloud run services update --update-env-vars` (revision `…00326`);
   the public count endpoint flipped to `enabled:true`.
4. **3 Cloud Scheduler jobs** created + ENABLED, mirroring the live `halatuju-application-reminders` job exactly (URI
   `…/internal/cron/<job>/`, `X-Cron-Secret` header, `Asia/Kuala_Lumpur`): F3 realtime (hourly), F3 digests (weekly Mon
   09:00), F4 purge (daily 03:00). The purge job was force-run as a smoke (safe no-op) — success.
5. **Live smoke.** New F9a/F4 endpoints return 401 (clean auth gate, **not** 500 — new code + new schema consistent for
   real requests, lesson #96); web `/sponsor` 200.

## What Went Well

- **Migrate-first held perfectly.** Applying additive schema to prod *before* the code deploy meant the live old code
  never saw a missing column, and the new tables were inert until the new code arrived — zero downtime, no 500s. The
  post-deploy 401-not-500 smoke confirmed consistency.
- **RLS in the same transaction as CREATE TABLE** closed the only real exposure window. Because Supabase serves the
  public schema over PostgREST, a new table without RLS would be readable via the anon key; creating + enabling RLS
  atomically (on empty tables, pre-deploy) meant PII was never reachable.
- **Pre-flight verification paid off.** Confirming prod was at scholarship `0048`, courses `0052`, that none of the 6 new
  tables existed, and that PKs use IDENTITY — all *before* writing — meant the DDL applied first-try, six for six, with
  no surprises.
- **Everything mirrored an existing, proven artefact.** The scheduler jobs copied the live reminders job; the flag is the
  documented env var; the migration approach is the established MCP + contenttypes-workaround pattern. Nothing was
  invented at the riskiest moment.

## What Went Wrong

- **Nothing broke, but the go-live shipped on DRAFT consent text — a deliberate, owner-accepted risk, not an accident.**
  *Decision:* the owner chose to go live now with `CONSENT_VERSION = 2026-draft-5` rather than wait for the lawyer, with
  refinements (and possibly flow changes) to follow. *Why it's acceptable:* real-user exposure on day one is minimal (1
  sponsor, 0 published students), the version string stays `draft` so a later bump cleanly re-attests everyone, and the
  programme can be re-darkened with one `--update-env-vars` if needed. *Follow-up obligation (tracked):* sync the
  lawyer-vetted wording + bump the version + re-attest; Tamil refine (TD-091/094/096/097/105/108). The risk is logged and
  owned, not hidden.

## Design Decisions

- **Go live on draft consent, refine post-lawyer** (owner, 2026-06-09) — logged in decisions.md.
- **RLS enabled atomically with each CREATE TABLE**, deny-by-default / service-role-bypass, matching every existing prod
  table — not a follow-up step (it was a deploy-blocking obligation, TD-093/098/100/102/106).
