# Engineering Lessons — HalaTuju

Cross-cutting lessons from sprint retrospectives. Only items that affect future work regardless of feature area.

- Bulk data loading via MCP tools should always be delegated to subagents to avoid context exhaustion. Each batch consumes significant context; 20+ batches will overflow the main window. (STPM Sprint 3)
- When modifying grade scales or lookup tables, always run golden master tests immediately — parsed data may contain legacy values not visible in the UI-facing code. (STPM Sprint 5)
- Use constants for localStorage keys instead of string literals to prevent key mismatch bugs across pages built in different sprints. (STPM Sprint 5)
- Always check `class Meta: db_table` in Django models before writing raw SQL against Supabase — default naming (`app_model`) doesn't apply when custom table names are set. **DANGER (v2.21.0): a same-purpose LEGACY table can also exist** — `StudentProfile` is `api_student_profiles` (617 rows), but a legacy Streamlit `student_profiles` (30 rows) is still there. An `ALTER ... ADD COLUMN` against the wrong one **SUCCEEDS silently** (no error — the column just lands on the dead table), so "success" is NOT confirmation. After any migrate-first `ALTER`, verify the column on the table the model actually uses AND sanity-check that a known existing column is present (e.g. the altered table should already have `stream_subjects`). This recurred despite two prior lessons — knowledge wasn't the gap, the check-at-write-time discipline was. (STPM Sprint 6; recurred + sharpened v2.21.0)
- For bulk data loads >500 rows into Supabase, generate a SQL file and use `psql` or a management command instead of MCP batches — each batch consumes context and 20+ batches will overflow the window. (STPM Sprint 6)
- Cloud Run `NEXT_PUBLIC_*` env vars are baked at build time — tagged revisions still call the main API URL unless the frontend is rebuilt with the tagged URL. (STPM Sprint 6)
- Django management commands must read config from `django.conf.settings` first, not `os.environ.get()` — Django doesn't auto-load `.env` into the environment. (STPM Sprint 7)
- For AI taxonomy classification, use a closed set of categories (no "add new if none fits") or include a two-pass approach — open-ended prompts produce hyper-specific values that defeat the purpose of a taxonomy. (STPM Sprint 7)
- Django management commands that modify data must verify the target database before running — print the database host at startup and abort if it's SQLite when production was intended. (STPM Sprint 8)
- When debugging hydration issues with Playwright, always wait for network idle before taking snapshots — pre-hydration snapshots produce misleading "not working" results. (STPM Sprint 8)
- Do not generate data on-the-fly in API responses when it should be a database entry — synthetic entries cause search gaps, badge inconsistency, and separate code paths that diverge over time. (Pre-U Sprint)
- Before inserting data into the DB, check how downstream pages render it — adding bare records can degrade rich UIs that used a different data source. (UI Polish Sprint)
- Always verify key/value mappings against actual API responses, not visual references — uppercase/lowercase mismatches cause silent display failures. (UI Polish Sprint)
- When cross-referencing CSV data with database records, never trust a single column for critical filters (e.g. bumiputera) — verify against a second source file or the institution itself. (Data Integrity Sprint)
- When matching courses across systems with different ID schemes, use name-based matching as fallback — institution types may use entirely different code conventions (FB/FC vs POLY-DIP/KKOM-CET). (Data Integrity Sprint)
- Tech debt audits must include test execution analysis — check pytest output for skipped/xfailed tests, not just code and docs. A test that silently skips is worse than a missing test because it creates false confidence. (Test Health Sprint)
- When migrating data sources (e.g. CSV → DB), immediately update or delete all tests that depend on the old source — `skipIf` guards on file existence mask the problem indefinitely. (Test Health Sprint)
- For server-rendered portals (MOHE, polycc, etc.), HTTP status codes are useless for link validation — always check rendered page content with a headless browser. (External Links Sprint)
- Never guess institution URLs from naming patterns — scrape the authoritative listing page instead. Even within a single system (e.g. IPG), URL conventions are inconsistent. (External Links Sprint)
- Design docs for rename/removal tasks must include a mechanical `grep` for all references before finalising the file list — manual estimates undercount because symbols propagate into tests, views, and utility code. (Subject Key Unification Sprint)
- Before changing a function signature, grep all call sites first — adding a new positional parameter breaks callers that pass options as the second argument. Use options objects for new optional parameters. (Saved Courses Sprint 1)
- When removing `as any`, check all types it was masking — a single `as any` often papers over multiple distinct type mismatches. Run `tsc --noEmit` after removal to catch cascading issues. (Quick Wins Sprint)
- When implementing a tech debt item, read the "What consistent looks like" field first — it defines the acceptance criteria. Skipping it leads to incomplete fixes that need a second pass. (Frontend Cleanup Sprint)
- When adding content for a trilingual project, always use the i18n system — never add content to DB fields that only support 2 languages. Check how many languages the project supports before choosing the storage method. (Tech Debt Quick Wins 2)
- Before writing the first frontend test in a project, verify that Jest is configured for the language (TypeScript needs `ts-jest` transform + path aliases in `jest.config.js`). An installed `jest` package is not enough — the config must exist. (IC Gate Sprint)
- When reusing a classifier or mapper across data sources, verify the semantics of each input column — same column name does not mean same data granularity. STPM `field` == aggregate `category`, unlike SPM where `field` is a specific sub-discipline. (Field Taxonomy Sprint 2)
- When building tools that process pipeline output, always run against real data before marking complete — synthetic test data can have different shapes (e.g. list of strings vs list of dicts) that only surface with production data. (Pipeline Sprint 3)
- Before creating migrations on a feature branch, always check `max(migration number)` on main — parallel branches creating same-numbered migrations is a recurring issue (third time in this project). Rebase or check before numbering. (Admin Auth Sprint)
- When accessing Stitch designs, use MCP `get_screen` instead of preview URLs — Stitch previews are not publicly accessible and will 404 in Playwright. (Identity Verification Sprint)
- Mark completed sub-tasks explicitly before moving on in long conversations — context loss causes duplicate work and user frustration when previously fixed code is "investigated" again. (Identity Verification Sprint)
- Never apply a fix pattern from one formula to another without independently verifying the target formula is actually wrong — "do the same for X" does not mean the same fix applies. Read the documentation block first. (Post-Launch Hardening Sprint)
- Display logic (how many items to show in a section) belongs in the frontend, not the backend — backend should rank and return a complete sorted list, not pre-split for a specific UI layout. (Post-Launch Hardening Sprint)
- For long-running subagent tasks (scrapers, deploys), verify `git log` before and after — concurrent subagents on the same branch can commit unexpected work while the main task is waiting. Use worktrees for isolation. (Pipeline Completion Sprint)
- When a user reports an "intermittent" bug, compare the failing path to a working path before hypothesising infrastructure causes — the differentiator reveals the actual bug. (i18n & Bug Fixes Sprint)
- Enforce security invariants (like "must have NRIC") via middleware, not per-endpoint checks — a single enforcement point prevents leaks when new endpoints are added and avoids the "check every time we implement a new feature" problem. (NRIC Hard Gate Sprint)
- After subagent work on frontend TSX files, always run `next build` before marking complete — TypeScript/JSX errors (like block-scoped variable ordering) are only caught by the compiler, not by reading the code. (NRIC Hard Gate Sprint)
- When a Django model field stores enum-like strings ("Ya"/"Tidak"), prefer BooleanField from the start — string booleans create type mismatches at serializer boundaries that are hard to diagnose. (i18n & Bug Fixes Sprint)
- localStorage is a cache, not a source of truth — always overwrite from the authoritative backend on login. Conditional writes (`if !localStorage`) cause stale-cache bugs that are hard to reproduce because they only affect returning users. (i18n & Bug Fixes Sprint)
- When fixing a data-type bug at a system boundary, trace the full pipeline (DB → API → frontend → cache → consumer → API) before starting — fixing one segment reveals the next, leading to multiple deploys for what should be one fix. (i18n & Bug Fixes Sprint)
- When applying Django migrations via raw SQL to bypass `InconsistentMigrationHistory`, verify the actual column types match the model AFTER applying — inserting into `django_migrations` only records the migration, it does not execute the DDL. A VARCHAR column read by a BooleanField silently coerces non-empty strings to `True`. (i18n & Bug Fixes Sprint)
- When adding a useEffect to a component with early returns (`if (x) return null`), place ALL hooks before ALL early returns — React's Rules of Hooks require hooks to be called in the same order on every render. A useEffect after an early return will crash when the condition changes. Spec reviewers should verify hook placement relative to conditional returns. (Auth Flow Canonical Refactor Sprint)
- Before drawing a data-driven conclusion that changes architecture (e.g. "use WhatsApp, not email"), verify the **authoritative** source — how the live system actually captures the field — not a convenience/back-office export. A blank or "N/A" column in an admin spreadsheet is not evidence of absence. (B40 Sprint 1)
- The frontend Jest config runs in **node** env (no jsdom, no React Testing Library). Put testable logic in pure modules (`src/lib/*.ts`) and unit-test those; don't write tests that render components — they have no DOM to render into. (B40 Sprint 2)
- When a sprint changes behaviour in an earlier sprint's code path (e.g. intake now also runs shortlisting), update that earlier sprint's tests in the same sprint, and run the **full** suite — a new-tests-only run hides the drift in the older tests. (B40 Sprint 3)
- Access an optional reverse OneToOne (e.g. `application.funding_need`) with `try/except Model.DoesNotExist` — Django's `RelatedObjectDoesNotExist` is **not** an `AttributeError`, so `getattr(obj, 'rel', None)` does not catch it and the access raises. (B40 Sprint 4a)
- The MANDATORY "prototype UI in Stitch before coding templates" rule applies to a multi-sprint build too, not just one-off pages — building the entire applicant + admin frontend across sprints without a single Stitch screen is the same miss at larger scale. Generate and get visual approval on the key screens before the first TSX file, even when the sprint plan reads as "backend + frontend together." (B40 Phase 1)
- When a field has two homes (e.g. financial data on both the application and the profile), name the single source of truth explicitly in the model docstring and make every other place read it live or snapshot it immutably — never store a second mutable copy. Two mutable copies of one fact have no defined winner and silently diverge. (B40 Phase 1.5a)
- Don't conclude a comms channel is dead from a back-office export — verify how the live system captures the field. ~600 students had verified Gmail (Google Sign-In) even though an admin spreadsheet's Email column was blank. (B40 Sprint 1)
- Keep all email/SMTP config env-driven from day one (`EMAIL_HOST`/`EMAIL_HOST_USER`/`EMAIL_HOST_PASSWORD`/`DEFAULT_FROM_EMAIL`) — a personal Gmail hardcoded as the sender is both a deploy-time secret leak risk and a deliverability/ownership problem. Default to the provider relay (Brevo) and verify the sender domain. (B40 Phase 1.5a)
- A DRF serializer reused across endpoints (e.g. `ProfileUpdateSerializer` for both `PUT /profile/` and `/profile/sync/`) — toggling a field to `read_only` or changing its validation changes EVERY endpoint that uses it. Enumerate the consumers before editing (here, making `nric` read-only correctly closed the PUT *and* sync write gaps in one change). (B40 Redesign Sprint 7)
- A pure-function engine's hand-built test fixtures (`SimpleNamespace`/dicts) silently return `None` for any attribute the engine reads but the fixture omits — so adding a new engine input (e.g. `household_size`, `upu_status`) won't fail loudly in the old fixtures, it just scores wrongly. When you change an engine's inputs, rebuild its test fixture to include every attribute it now reads. (B40 Redesign Sprint 8)
- In a multi-step / tabbed form, render submit + validation errors at **form level**, not inside one step's panel. If validation can send the user to a different step (e.g. "jump to the tab with the error"), an error message scoped to the last step is invisible exactly when it's needed. (B40 Redesign Sprint 9)
- Don't probe a file path with a chained `ls A || ls B 2>/dev/null` — the failing branch is swallowed and the surviving branch's output doesn't say which path it came from, so you can mis-attribute the result (cost a wrong-path assumption this sprint: i18n is in `src/messages/`, not `src/i18n/`). Run one explicit check per candidate, or echo the resolved path. (B40 Redesign Sprint 9)
- Make browser-storage helpers (localStorage/sessionStorage) **storage-injectable** (`fn(..., storage?: StorageLike)` defaulting to a `safeSession()`/`safeLocal()` that returns null when undefined) — then node-env jest can unit-test the full round-trip with a Map-backed fake, and the helper is SSR-safe for free. (B40 Redesign Sprint 9b)
- A persistent flag (localStorage/sessionStorage) set **before** a multi-step flow goes stale if the flow is abandoned — there's no clean "cancelled" boundary to clear it, so a later unrelated run reads the stale flag. Prefer threading the intent through navigation (query param) over a persistent flag; if you must use a flag, clear it on every legitimate entry/exit boundary and accept the residual abandon edge. (B40 Redesign Sprint 9b)
- Kill background `next dev` servers **by listening port** (`netstat -ano | findstr :PORT` → `taskkill /F /PID`), not `pkill -f "next dev"` — on Windows the detached `next-server` child survives `pkill` and stays bound across turns (causes `EADDRINUSE` on restart, and a stale server serves the OLD bundle so screenshots silently go wrong). Confirm the restart actually bound before navigating. (B40 Redesign Sprint 10)
- Don't assume "push triggers deploy" also runs DB migrations — **verify the deploy actually migrates** before pushing. HalaTuju's Cloud Run triggers are auto-generated build→push→deploy with **no migrate step** (the Dockerfile CMD is just gunicorn), so shipping code that needs new columns would 500 for live users. Apply additive migrations **migrate-first against prod** (old code keeps working), *then* push `main`. (B40 Redesign Sprint 12b)
- Django `AlterField` that only changes a column's **default** does NOT rewrite existing rows — pre-existing rows keep their old value (newly *added* fields do pick up the new default). After changing a default on an existing column, **explicitly sync the existing config rows** (this left the prod cohort on the old 5/3.0 thresholds after the engine was changed to 4/2.9). (B40 Redesign Sprint 12b)
- A safety classifier blocking a prod action (e.g. pulling prod DB creds to migrate) is not a thing to engineer around — **surface it to the user and let them choose** (approve, switch mode, or run it themselves). Honest stop > clever workaround. (B40 Redesign Sprint 12b)
- Running `next build` (production) and then `next dev` over the **same `.next` directory** corrupts the dev chunk graph — the page server-renders fine (screenshots look OK) but the client throws `ChunkLoadError: Loading chunk app/layout failed` and never hydrates, so nothing is interactive. The first dev hit also times out *during* its 10–13 s compile. Before a dev-server preview that follows a build, `rm -rf .next` and let dev rebuild from scratch; give the route one warm-up navigation before interacting. (B40 plans-redesign P2)
- When verifying a prod migration via **raw SQL**, read the model's `Meta.db_table` FIRST — never assume Django's default `<app>_<model>` table name. The scholarship models use custom `db_table` (`scholarship_applications`, not `scholarship_scholarshipapplication`), so a verification query against the default name reported "0/7 columns / table does not exist" mid-deploy — a false alarm. Verify through the ORM/`dbshell`, or grep `db_table` in models.py before querying `information_schema`. (B40 plans-redesign P5 ship)
- On the HalaTuju prod DB, `manage.py migrate` can **exit non-zero on a fully successful migration**: it prints "Applying … OK" then errors `relation "django_content_type" does not exist` from the `post_migrate` signal (create_contenttypes/create_permissions). Prod has **no contenttypes/auth tables** (never created), so that signal always fails — but the additive `ADD COLUMN`s have already committed + recorded in `django_migrations`. A non-zero exit from the post_migrate step is NOT a migration failure; verify the actual columns + the `django_migrations` row directly. (B40 plans-redesign P5 ship; see TD-058)
- A nullable/optional serializer field sent as an **explicit `null`** can still 400 a serializer that only tolerates the field being **omitted** — null and absent are not the same to DRF. If the client may send `null` for an optional field, strip nulls in `to_internal_value` (or mark it `allow_null`). Cost this sprint: the apply page POSTed `coq_score: null` for 100% of profiles → 400 → swallowed into an empty pathway dropdown. (Post-launch apply polish, 2.2.1)
- Normalise enum-like inputs (gender/nationality → the engine's Malay canonical `Lelaki`/`Warganegara`) at **every** engine entry point, not just the one serializer that happens to do it. A second caller (the STPM view) passed raw `male`/`malaysian` straight to the engine and silently returned **0 eligible** for every STPM student. Extract a shared normaliser and call it at the boundary, not per-view. (Post-launch apply polish, 2.2.4)
- A **one-time prefill effect** that locks on the *first* profile snapshot misses fields that arrive later. A brand-new user claims their NRIC at the auth gate **after** the page mounts, so the early snapshot has no NRIC; locking the prefill then left the field blank forever. Gate a one-time prefill on the **awaited field's presence** (`profile.nric`), not merely "the profile object exists". (Post-launch apply polish, 2.2.7)
- Chrome **ignores `autocomplete="off"`** for inputs it heuristically classifies as address/contact and will pop its saved-address list over a custom combobox's own menu. Use `autocomplete="new-password"` on type-to-search comboboxes to suppress it (text input → no password UI), plus `data-1p-ignore`/`data-lpignore` to mute password managers. Note: this can't be reproduced in headless Playwright (no saved addresses) — apply the known-good fix and have the user confirm on their real browser. (Post-launch apply polish, 2.2.6)
- For an **additive** migration on the HalaTuju prod DB, applying the DDL + recording the `django_migrations` row via the Supabase MCP (`execute_sql`, one transaction) is cleaner than `manage.py migrate` here: it sidesteps TD-058's non-zero `post_migrate` exit and the session-pooler/CRLF credential juggling, and you verify the columns + the migration row in the same connection. Replicate Django's DDL exactly — for a `CharField(default='')`, `ADD COLUMN … NOT NULL DEFAULT ''` (backfills existing rows) then `DROP DEFAULT` (Django keeps the default app-side). Only for additive changes; a model-creating migration still needs the contenttypes/auth fix (TD-058). (Post-launch apply polish, 2.3.0)
- Stitch `generate_screen_from_text` **times out client-side on content-dense screens** and a timed-out generation does **not** reliably persist (polling `list_screens` after shows nothing). Don't sit in a poll/retry loop and don't block the sprint on it: generate **one** representative screen to get the *pattern/layout* signed off (that can cover several sprints that reuse it), switch to `GEMINI_3_FLASH` for speed, and prototype the remaining sections **just-in-time before their own sprint** rather than all up front. (Application-redesign S1)
- Under heavy main-thread context, delegate a **contained, well-specced** sprint to a single subagent (fresh context) and have the orchestrator review the diff + screenshots + re-run `next build` before deploying — keeps the sprint within budget without losing the verify/deploy gate. Spec must say "no commit/push/deploy; leave changes in the tree" so deploy stays with the orchestrator. (Application-redesign S1)
- Never run `next build` (or a second `next dev`) against a project's `.next` **while a subagent is doing a dev-server/Playwright screenshot in the same project** — both writing `.next` triggers the build-vs-dev corruption (a bogus `PageNotFoundError: /_document` + non-zero exit, despite "Compiled successfully"). Serialise: wait for the screenshot task to kill its dev server before building, or isolate the screenshot task in its own git worktree. (Application-redesign S2)
- When delegating build+screenshot work to a subagent, tell it to **SAVE screenshots to a named path and NOT delete them** (delete only the throwaway preview route + dev server) — otherwise the orchestrator can't review the UI and has to send a round-trip to regenerate. (Application-redesign S2)
- A **timed-out Stitch generation doesn't fail cleanly — it can persist LATER as a same-titled duplicate.** So when you poll `list_screens` after a timeout, the screen you find may be an OLD attempt (e.g. a pre-revision design), not your latest prompt — verify the rendered CONTENT matches what you just asked for before showing it for sign-off; don't trust the title or assume the first/newest-listed screen is yours. (Application-redesign S3)
- A content-dense Stitch screen still times out on `GEMINI_3_FLASH` when **every repeating element is fully specced** — for a list/repeat-heavy screen, describe each repeating element **once** and summarise the rest ("…and three more rows: X, Y, Z") so the render stays light enough to persist. The trimmed prompt succeeded first-try where the fully-enumerated one (8 cards × full helper text + states + bottom bar) had timed out and not persisted. (Application-redesign S4)
- When adding a new **external-SDK module** (Cloud Vision, Gemini, Stripe, any vendor whose Python or Node package gets a `from vendor` / `import vendor`), the **`requirements.txt` (or `package.json`) bump is part of the same diff as the code** — not a follow-up. Forgetting it doesn't break the build (lazy imports + graceful-degradation just say "module not installed" at runtime) but it does cost an avoidable second deploy when caught later. Cheap catch at push time: after any new `from <vendor>` line, grep `git diff --stat` for `requirements.txt` (or `package.json`) and fail loud if absent. (S13)
- When introducing a feature behind a **billable external API**, ship the code with the API **disabled** first (graceful-degradation path: `ImportError` / "AI service not configured" → soft fallback), then enable the API as a **separate explicit step** after the user's cost sign-off. Lets the code/tests/migration land safely; the fallback path is exercised in production for free; teething surfaces there before any billable call. Then flip the switch + one cheap public-sample-image smoke (e.g. `gs://cloud-samples-data/...`) to prove project-level provisioning before asking the user to upload real data. (S13)
- **Never pipe `npm run build` (or any pass/fail gate) to `grep` and read the pipeline's exit code as the build's.** The pipe's exit code is the *last* command's — grep's, not npm's. When npm exits 1 with a TypeScript/lint error, grep happily matches the `Failed to compile`/`Type error` lines and returns 0, and you read "EXIT=0" as "build green." This masked **two** broken web pushes for TD-059 (deploy-budget breach: 3 pushes instead of 2). Either `set -o pipefail` at the start of the shell, or check `${PIPESTATUS[0]}` explicitly, or capture to a file (`npm run build > build.log 2>&1; echo "EXIT=$?"`) and read the unmasked exit code separately from any greps. (TD-059)
- **When changing a shared model's shape, grep ALL parallel type definitions before declaring done.** This project keeps two frontend interfaces for `FundingNeed`: `lib/api.ts` (student-facing) and a separately-typed `lib/admin-api.ts AdminScholarshipDetail.funding_need` (admin-facing). Updating only the first broke `next build` on the admin page (`Record<string, number | string>` left `categories?.length` typed as `string | number` → no `.length`). Before pushing a model-shape change, `grep -rn "<ModelName>"` across `lib/` and confirm every parallel definition matches. (TD-059)
- When a unit wraps an external call (AI/Gemini, HTTP, etc.), **extract the payload/prompt builder as a pure function and unit-test it directly** — don't rely only on tests that mock the *whole* call. The AI sponsor-profile generator referenced four model fields the profile-canonical refactor had removed and would have 500'd on a real "Generate", but the only test mocked the entire `generate_sponsor_profile`, so the broken `_build_prompt` was never executed and the schema drift failed nowhere (rotted silently across the refactor + several sprints, masked further by the dormant programme / no CI API key). A pure-builder test (`test_profile_engine.py`) makes input-schema drift fail loudly while the external call stays mocked. (Application-redesign S5c, TD-060)
- Don't run the **full backend `pytest` suite concurrently with `next build`** on this (8GB-RAM) machine. `apps/courses` loads the large course DataFrame at app init (`AppConfig.ready()`), and under memory/CPU contention from the Node build, `TestEligibilityEndpoint` setUp raises `ERROR` (not assertion failures) — 24 spurious errors that vanish (`24 passed`) when the class is re-run alone. Serialise the two, or run the eligibility tests separately. Treat a burst of **setup ERRORS confined to one heavy test class** as a contention smell, not a regression — re-run in isolation before believing it. (Application-redesign S5a)
- When proposing to **delete a field that has a confusingly-similar name to a kept field**, name the exact DB column + state both readers and writers in the proposal. "Drop the phone input" is ambiguous when there are two phone columns; "drop the dead `phone` column (no readers/writers outside /profile); keep `contact_phone` (canonical, /apply ⇄ /profile)" is not. The user reasonably pushed back on the ambiguous framing — would have been faster to disambiguate the first time. (S14)
- For an opinionated default (e.g. "auth email *is* the contact email unless the user says otherwise"), prefer a **read-time fallback in the API** over a **one-shot DB backfill**. The backfill mutates rows the user didn't choose, irreversibly merging "they actively chose this" with "we wrote it for them" — and locks the policy. A read-time fallback is one line to change later, naturally handles new users (phone signups have no auth email to fall back to), and lets admin SQL distinguish "explicit" from "implicit" by inspecting the empty column. Used for `contact_email` in S14 (574 rows untouched). (S14)
- **Unit tests on a parsing/OCR heuristic are necessary but not sufficient — the heuristic also has to be validated against ONE real document before deploy.** Synthetic test fixtures encode my assumptions about the document's structure (where the state line sits relative to postcode; what a "name line" looks like). On a real MyKad the assumptions are subtly wrong (state sits BELOW postcode in Vision's reading order; "TAMAN SEMANGAT" looks identical to a one-word name under "all-caps no-digit" filtering). S15's Vision address extraction took **3 deploys** to converge because each iteration's tests passed but the real-document output was wrong. **Practical fix for the next OCR heuristic:** before shipping, ask the user to send a real-document screenshot, dump the OCR text for it, and validate the heuristic against THAT text — not just synthetic samples. The same protocol catches the same class of bug for ANY parser-from-real-world-input (PDFs, scanned forms, screenshots, etc.). (S15)
- **In plan mode, the Explore brief must explicitly verify model location for any migration-class task** — not rely on my mental model of which model the field belongs to. S15 brief said `siblings_studying` lives on `StudentProfile`; it actually lives on `ScholarshipApplication`. The Plan agent caught it before any code was written. Cost was a re-write of the migration name + scoped target. The fix is procedural: when scoping a column add/drop/move, the Explore agent prompt must include "confirm which model the field currently lives on by reading the model file" as an explicit step. The Explore agent confirmed surrounding context but didn't verify the model location because I didn't ask. (S15)
- **When a user-facing list looks "too short", grep for a SECOND copy of that list on the other side of the wire before assuming the UI is the only fix.** The SPM stream dropdowns were under-populated (frontend `subjects.ts` pools), but the merit engine kept its OWN hardcoded copy of the same pools (`engine.py`), and the 30% stream weight was computed from the backend copy. Fixing only the frontend would have let students pick subjects that silently scored on the 10% elective weight instead of 30% — a worse, invisible bug than the original. Whenever the fix is "add items to a list that drives behaviour", ask: does any other layer (engine, serializer, validator, report) hold a parallel copy that must move in lockstep? If yes, change both in the same commit and add paired tests asserting equal membership/size. (S18)
- **`status='submitted'` ≠ "user is at /application Step 4."** The decision engine scores silently at submit and the applicant stays at `submitted` until the +48h scheduler fires; only `shortlisted` applicants see the post-shortlist Step-4 tabbed UI (Quiz/Story/Funding/Documents/Consent). Pre-reveal applicants see the "received" status card. **Before scoping any change that affects the /application Step-4 UI, run `SELECT status, COUNT(*) FROM scholarship_applications GROUP BY status` to see who actually sees the affected UI today** — not just `WHERE submitted_at IS NOT NULL`. In S19 I mis-framed a "parent_ic universal compulsory" change as having retroactive impact on 12 submitted applications; user pushed back, and verifying showed all 12 were pre-reveal — the change was forward-looking, not retroactive. The wrong framing would have caused unnecessary worry / unnecessary work. (S19)
- **When the user names a UX/convention explicitly** ("let's be consistent across the forms"), encode it in CODE not just docs — a tiny component (`InfoBox` in S19; `FieldLabel` extracted in S15) is the cheapest enforcement mechanism and self-documents via prop types. Inline `className` strings + a comment are forgettable; a typed component is not. (S19)
- **`for…of set` triggers Next.js downlevel-iteration TS errors** on this project's tsconfig target. Prefer `Array.from(set).every(…)` or `.some(…)` for Set/Map iteration in `.tsx` files — same intent, no downlevel flag needed. (S19)
