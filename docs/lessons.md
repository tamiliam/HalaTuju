# Engineering Lessons — HalaTuju

Cross-cutting lessons from sprint retrospectives. Only items that affect future work regardless of feature area.

- **Reusing a "once per X" component inside a per-item list needs dedup at the LIST level, not the item level.** V6 mounted the per-*earner* `IncomeClusterCoach` on every open income doc-task, so a member with two tasks (IC + slip) would render two identical coaches. The component is anchored once per cluster on its native surface (`clusterAnchorKey`); a per-item list has no such anchor unless you build one. Fix: compute a per-key anchor (first item of each key) in the list and pass a `show…` flag so only the anchor renders it. When you lift a cluster/summary component into a flat list, ask "how many times will this mount?" and dedup by its natural key. (Verification-model V6, 2026-07-04)

- **When you add a gate/precondition to a SHARED endpoint, grep the whole test tree for every caller — not just that endpoint's own test file.** The V5 QC gap floor (refuse QC-Accept while a verdict fact is red) broke `test_decision_reopen.py`, not `test_qc_gate.py`: the reopen suite drives a real QC-Accept on a document-less fixture whose live `build_verdict` is all-gaps, so the floor 400'd it. I re-pinned the QC-gate tests but never considered that other endpoints' tests exercise the same POST. Before shipping a new precondition, `grep -r '<endpoint-path>' tests/` and audit each caller — a fixture that was fine under the old contract can silently violate the new one. (Verification-model V5, 2026-07-04)

- **A status/state-machine side-effect must fire from the event that MEANS the new state — not from incidental activity that merely co-occurs with it.** `profile_complete → interviewing` was advanced on any interview *draft save*; when V3 folded the agenda into the draft, early triage (even an agenda-item delete) started saving drafts, and four live apps silently flipped into the interview funnel with no accountable owner. The trigger was wrong, not just under-guarded. When a UI change makes an existing event fire more often (or in new contexts), audit every state-advancing side-effect keyed to that event and re-anchor it to the action that actually denotes the transition (here: proposing times / submitting — plus an assigned owner). Invariant-test the target state (`interviewing ⇒ assigned AND (slots OR submitted session)`) so an incidental writer can't satisfy it. (Interview-trigger hotfix, 2026-07-03)

- **Multi-line source edits through a bash-heredoc-wrapped python string mangle escapes (`
` became a real newline → a SyntaxError written INTO the file). Write the edit script to a scratch file and run it, or use the editor tool — and always run the affected tests immediately after a scripted edit** (the collection error caught it in seconds). (Code-health S5, 2026-07-03)

- **A known incident mode documented in a memory note is still an OPEN bug — encode the protection as a code invariant.** "Never re-extract locally (no Storage → wipes vision_fields)" lived as a memory rule for weeks while every writer stayed happy to overwrite good data with an empty error read; one absent-minded run away from data loss. The fix (clobber guard: a FAILED run never replaces a prior good read of the same immutable blob) took ~20 lines. When a post-incident rule says "never do X because Y destroys Z", ask what code change makes Y unable to destroy Z. Corollary for tests: when a command infers success from a side-effect (e.g. a timestamp advancing), its tests' mocks must produce that side-effect. (Code-health S2, 2026-07-03)

- **A cross-repo "mirror of X — keep in sync" comment is not a mirror; pin it with a test that reads the other repo's file and fails loudly (never skips).** `academic_engine._SUBJECT_BM` drifted 64 keys behind `subjects.ts` in one month because the sync was enforced by a comment; both sides stayed self-consistent so nothing failed — only real students hit the "missing subject" loop. The drift test must also sanity-check its own parse (min key count), or a restructure of the source file silently turns it into a 0-missing no-op. (Code-health S1, 2026-07-03)
- **When two different outward events share one stamp field, the consumer that needs to distinguish them can't — give each its own stamp.** `decision_email_sent_at` meant both "shortlist PASS email went" and "decline email went"; `cancel_pending_decline` read it as the latter, so for every normally-processed student the cancel silently did nothing. A status-machine test must also ENTER states via the real transitions (the old test built the fixture directly in the target status, skipping the very step that stamps the field, and pinned the bug as green). (Code-health S1, 2026-07-03)

- **Before enabling a cron that auto-acts on a CLASS of existing records (emails, charges, status flips), backfill the existing rows as "already handled" in the SAME migrate-first step — or the first run blasts them all.** Switching the award email to a cool-off cron would have re-emailed every already-notified award (and the embargoed one) on its first tick; the fix was to backfill `offer_emailed_at=now()` on all live awards before the scheduler existed. When you add a "do this to everything matching X" job, the existing matches are the danger, not the new ones. (Award email cool-off, 2026-06-29)

- **The sponsor pool DETAIL serves `SponsorProfile.anon_markdown`, not `final_markdown` — they are two fields that only the approve-path keeps in sync.** Publishing a profile (flipping `anon_published`) does NOT refresh `anon_markdown`, so a student published outside the normal approve flow (e.g. a manual/SQL publish, or one whose final was regenerated after the last publish) shows sponsors a STALE profile. When publishing/regenerating, mirror the current `final_markdown` → `anon_markdown` (+ `anon_model_used`/`anon_generated_at`) and regenerate the card `anon_blurb`, exactly as `AdminRecordVerdictView`'s approve+finalise branch does. (Sponsor self-funding, 2026-06-29)

- **`manage.py sqlmigrate` renders for the LOCAL backend (SQLite) — never paste it into a Postgres migrate-first.** Local dev is SQLite, prod is Postgres; `sqlmigrate courses 0059` printed a full SQLite table-rebuild (`CREATE new__…; INSERT…SELECT; DROP; RENAME`) for what is a one-line `ALTER TABLE … ADD COLUMN` on Postgres. For migrate-first via the Supabase MCP, **hand-write the Postgres DDL** and verify it against the live schema (`information_schema.columns`, an existing sibling table's id/timestamp/RLS conventions) before applying. (WhatsApp go-live, 2026-06-21)
- **Parallel feature branches each grab the next migration number blindly — check `max(migration)` on `main` before merging any of them.** doc-slots and whatsapp-comms both created `scholarship/0067` (neither could see the other off the same base). The later merge must renumber (→`0068`) and re-point its `dependencies`. Make "diff the migration leaves vs main" a pre-merge step whenever >1 branch is in flight. (WhatsApp go-live, 2026-06-21)
- **Split go-live config into agent-settable (non-secret) and owner-settable (secret) so secrets never transit chat.** WhatsApp go-live: the agent set `TWILIO_WHATSAPP_FROM` + the template SID via `--update-env-vars`; the owner pasted `TWILIO_ACCOUNT_SID`/`TWILIO_AUTH_TOKEN` into the Cloud Run console. Ship dark, set non-secrets, owner adds secrets, then flip one `*_ENABLED` flag after a smoke test (check blast radius first). (WhatsApp go-live, 2026-06-21)
- **A "fire when within X" reminder/window with no lower bound fires instantly on a late trigger — gate on the notice the trigger actually gave.** The interview reminder ("within 24h", "within 1h") had no floor, so a same-day booking got an instant "24h reminder." Fix: gate on `event_time − created/booked_at` (the notice), keeping firing late-tolerant so cron jitter never skips a legit send. Applies to any scheduled-nudge feature. (Interview reminder-notice, 2026-06-21)

- **A server-side validator must accept EVERY state the UI can legitimately produce — or a save silently 400s and the user's input is LOST.** `_validate_findings` rejected `verdict=''`, but the cockpit's normal action (typing a finding without clicking a verdict button) sends exactly that; the whole Save-draft was rejected and the reviewer's notes vanished on reload. When you add/tighten a validation rule, enumerate the field's possible UI states — **especially the empty/default state** — and accept the legitimate ones. Bonus guard: a failed save should not look like a success; surface the error AND don't let a rejected write masquerade as saved. (Cockpit data-loss fix, 2026-06-20)
- **Any AI prompt that reasons about TIME must be given today's date — the model doesn't know it.** Gemini flagged a past event ("accident on April 3rd, 2026") as "in the future" because the gap-spotter prompt never stated the current date. Inject today's date (in the user's timezone — MYT here) into any prompt that may compare dates, with an explicit past-vs-future instruction. This is systematic: audit other prompts (`profile_engine`, `help_engine`, `vision`) before they reason about dates too. (Cockpit AI-date fix, 2026-06-20)

- **i18n parity only proves en==ms==ta — it does NOT prove a `t()` key EXISTS. A page can reference keys that were never added and ship silently, rendering the raw key path.** The sponsor redesign (R1–R4) referenced ~47 `sponsorPortal.{impact,journey,activity,…}.*` keys that were never added to any locale; the My Giving/Students/Account pages rendered literal `sponsorPortal.impact.totalGiven` etc. It went undetected for 4 sprints because parity passed (all three locales equally missing), `next build`/jest don't validate `t()` keys, and the surface was dark (no real user saw it). **Guard:** a test that scans source for static `t('ns.key')` calls and asserts each resolves in en.json (dynamic-aware: skip `${...}`/concat keys by matching only keys ending at a closing quote). See `halatuju-web/src/messages/__tests__/sponsor-i18n.test.ts` (R7) + the older `admin-scholarship-i18n.test.ts` (TD-120). A "dark/dormant" surface especially needs this — there's no live user to notice raw keys. (Sponsor Redesign R7)

- **Bind an irreversible/outward side-effect to the state transition it belongs to — not to a nearby earlier action that happens to be convenient.** Sponsor "publish" was stapled to the reviewer's Accept verdict, but that fires one step BEFORE the QC gate — so a student went in front of sponsors (visible AND fundable) while still awaiting QC. The fix wasn't a read-layer filter; it was moving the publish to the transition that actually means "cleared for sponsors" (QC Accept → `recommended`). When a side-effect leaks across a gate, ask "which button/transition is the true trigger?" and move it there, so the data model matches the state machine instead of being patched at read time. (Keep a belt-and-suspenders read gate too — here `status == 'recommended'` in both `is_pool_eligible` and `is_fundable`.) The owner had to reject two read-layer patches first ("not a solution — we need a clear transition") before this framing landed. (Publish-at-QC Sprint, 2026-07-02)
- **When you introduce a slot/uniqueness key, enumerate EVERY writer of that slot — not just the primary path.** The document slot key `(doc_type, household_member)` (TD-115) was correct for the apply form, where each `(type, member)` is unique per application. But the Action Centre lets a reviewer request several docs of the *same* type (multiple "Other", or the same income doc for a different person) — those writers were never considered, so each upload silently overwrote the last and live applicants lost documents (Theepicaa: 5 requested, 1 stored). Fix was a third key dimension (`request_code`). Before shipping any "this overwrites the existing one in the same slot" rule, list all code paths that create that row (apply form AND reviewer/officer requests AND system-gap auto-resolves) and confirm the key distinguishes the ones that must coexist. (Request-owned doc slots, 2026-06-21)
- **A hand-written `CreateModel` migration must use `BigAutoField` for `id` — the project's `DEFAULT_AUTO_FIELD` is `BigAutoField`.** Writing a migration by hand (e.g. to control a seed `RunPython`) and defaulting the PK to `models.AutoField` from memory leaves model-state drift: `makemigrations --check` then demands a follow-up `…_alter_<model>_id`. Copy the `('id', models.BigAutoField(auto_created=True, primary_key=True, serialize=False, verbose_name='ID'))` line from a recent `CreateModel` migration, and always run `python manage.py makemigrations <app> --check --dry-run` before committing a hand-authored migration. (Sponsor Redesign R5)

- **A button that "reverses" or "edits" a finalised thing must touch the same state the original action published — check what the existing control actually does before extending it.** The cockpit "Edit" on a recorded decision was frontend-only (`setEditDec`); it unlocked the form but left the student's profile LIVE in the sponsor pool. Renaming it to "Reopen" without wiring it to `SponsorProfile.anon_published` would have let a reviewer correct a decision while the old profile stayed visible to sponsors. Before building a reverse/edit flow, trace where the original action's effect LIVES (here: the pool-publish gate) and undo/redo that, not just the editor lock. (Decision reopen, 2026-06-18)
- **When a feature writes a per-PERSON quality metric, confirm the increment trigger with the owner before coding — the trigger point assigns blame.** "Reopen counts as a reviewer error" is ambiguous: count every reopen (A) penalises an exploratory reopen the super then cancels; count only saved-change reopens (B) doesn't. Surfaced the A/B fork with a recommendation rather than guessing; the owner chose B. Also: derive such a count from an audit LOG (`COUNT(resulted_in_change=True)`), never a bare counter — it stays reconstructable and carries the reason/who/when. (Decision reopen, 2026-06-18)
- **Never pass a secret as a `gcloud` flag value that can echo on error.** A wrong flag name (`--update-headers` on
  `jobs create` instead of `--headers`) made gcloud print the unrecognised-argument *with its value* — leaking
  `CRON_SECRET` into the transcript. Verify the exact flag first (`gcloud ... --help`), keep the secret in a shell var
  (never a literal), and prefer commands/paths that can't echo it. If one leaks: rotate it — update the env var
  (`--update-env-vars`, new revision) AND every consumer (all 11 Cloud Scheduler `X-Cron-Secret` headers), then confirm
  the OLD value now returns 403. (CRON_SECRET rotation, 2026-06-16)
- **Version any AI prompt and stamp the version on every artifact it generates.** Without it, a prompt redesign
  silently leaves old outputs in place and they're only caught by chance (the #18 stale-draft: an old-format profile sat
  in prod until the owner compared two side by side). Add a `PROMPT_VERSION`, store it on each generated row, and make
  regeneration version-aware (skip rows already on the current version, refresh the rest). Detect staleness by VERSION,
  never by date — and remember an idempotent "has output → skip" sweep will NOT refresh stale rows after a prompt change.
  (Sprint 2026-06-16)
- **Sensitive-attribute redaction must reach the NARRATIVE, not just structured fields.** Hiding ethnicity in the grades
  (group subjects, don't name "Bahasa Tamil") isn't enough — the student's own words ("inspired by her Tamil teacher,
  preserving her mother tongue") leak it too. The fix that preserved meaning was to GENERALISE in the prompt (keep the
  motivation, drop the label: "her mother tongue", "a teacher who inspired her"), applied in the shared redaction block
  so both the draft and the final inherit it. (Sprint 2026-06-16)
- **Don't hand-maintain a PARTIAL lookup map parallel to a canonical one — it drifts and leaks the raw key.** A 15-entry
  `_GRADE_LABELS` missed `b_tamil`, so the raw key `B_TAMIL` was fed to the prompt and surfaced non-deterministically.
  Prefer mapping inputs to a small FIXED safe vocabulary (subject GROUPS) with a generic fallback ("other subjects"), so
  an unmapped input degrades gracefully instead of leaking. And when deleting an i18n PARENT object, verify each LEAF is
  unused — TD-118 deleted `finalProfile` whole when only 6 of 7 leaves were dead. (Sprint 2026-06-16)
- Verify **feature/flag state against the running system**, not against CLAUDE.md or a summary — both rot. State a flag is on/off only after `gcloud run services describe <svc>` shows the env var; state a feature is built/read-only only after reading the code on **`main`** (the worktree), never the primary checkout (which may sit on another agent's branch). Two wrong assertions this round (`/profile` "read-only", `CHECK2_STUDENT_QUERIES_ENABLED` "off") both came from trusting docs + a primary-checkout read. (Sprint 2026-06-15)
- When an Explore/search subagent reads the repo during multi-agent work, it reads the **primary checkout's branch** (possibly not `main`). Point it at the worktree path, or re-verify any load-bearing claim against `main` before reporting it. (Sprint 2026-06-15)
- A feature flag is NOT live just because the Cloud Run env var is set — it must be **read into Django settings** (`settings.X = os.environ.get('X', '')...`). `CHECK2_AUTO_GENERATE` was used via `getattr(settings, 'CHECK2_AUTO_GENERATE', False)` but never defined in any settings file, so it was permanently False and the handoff auto-draft silently never fired. When adding/toggling a flag: grep the settings package for its name; if absent, wire it before touching the env var. (Sprint 2026-06-15)
- This sandbox cannot reach the prod Supabase DB (DNS blocked — that's why schema changes go via the Supabase MCP). To run a one-off command that needs prod DB + an external API (e.g. a Gemini backfill), run it ON the service via the internal cron endpoint (a no-arg management command in `CronRunView.JOBS`) or a Cloud Run Job — not a local `manage.py` run. (Sprint 2026-06-15)

- Bulk data loading via MCP tools should always be delegated to subagents to avoid context exhaustion. Each batch consumes significant context; 20+ batches will overflow the main window. (STPM Sprint 3)
- When modifying grade scales or lookup tables, always run golden master tests immediately — parsed data may contain legacy values not visible in the UI-facing code. (STPM Sprint 5)
- Use constants for localStorage keys instead of string literals to prevent key mismatch bugs across pages built in different sprints. (STPM Sprint 5)
- Always check `class Meta: db_table` in Django models before writing raw SQL against Supabase — default naming (`app_model`) doesn't apply when custom table names are set. **DANGER (v2.21.0): a same-purpose LEGACY table can also exist** — `StudentProfile` is `api_student_profiles` (617 rows), but a legacy Streamlit `student_profiles` (30 rows) is still there. An `ALTER ... ADD COLUMN` against the wrong one **SUCCEEDS silently** (no error — the column just lands on the dead table), so "success" is NOT confirmation. After any migrate-first `ALTER`, verify the column on the table the model actually uses AND sanity-check that a known existing column is present (e.g. the altered table should already have `stream_subjects`). This recurred despite two prior lessons — knowledge wasn't the gap, the check-at-write-time discipline was. (STPM Sprint 6; recurred + sharpened v2.21.0)
- For bulk data loads >500 rows into Supabase, generate a SQL file and use `psql` or a management command instead of MCP batches — each batch consumes context and 20+ batches will overflow the window. (STPM Sprint 6)
- Cloud Run `NEXT_PUBLIC_*` env vars are baked at build time — tagged revisions still call the main API URL unless the frontend is rebuilt with the tagged URL. (STPM Sprint 6)
- Django management commands must read config from `django.conf.settings` first, not `os.environ.get()` — Django doesn't auto-load `.env` into the environment. (STPM Sprint 7)
- For AI taxonomy classification, use a closed set of categories (no "add new if none fits") or include a two-pass approach — open-ended prompts produce hyper-specific values that defeat the purpose of a taxonomy. (STPM Sprint 7)
- Django management commands that modify data must verify the target database before running — print the database host at startup and abort if it's SQLite when production was intended. (STPM Sprint 8)
- When debugging hydration issues with Playwright, always wait for network idle before taking snapshots — pre-hydration snapshots produce misleading "not working" results. (STPM Sprint 8)
- Do not generate data on-the-fly in API responses when it should be a database entry — synthetic entries cause search gaps, badge inconsistency, and separate code paths that diverge over time. (Pre-U Sprint)
- Before inserting data into the DB, check how downstream pages render it — adding bare records can degrade rich UIs that used a different data source. (UI Polish Sprint)
- Always verify key/value mappings against actual API responses, not visual references — uppercase/lowercase mismatches cause silent display failures. (UI Polish Sprint)
- When cross-referencing CSV data with database records, never trust a single column for critical filters (e.g. bumiputera) — verify against a second source file or the institution itself. (Data Integrity Sprint)
- When matching courses across systems with different ID schemes, use name-based matching as fallback — institution types may use entirely different code conventions (FB/FC vs POLY-DIP/KKOM-CET). (Data Integrity Sprint)
- Tech debt audits must include test execution analysis — check pytest output for skipped/xfailed tests, not just code and docs. A test that silently skips is worse than a missing test because it creates false confidence. (Test Health Sprint)
- When migrating data sources (e.g. CSV → DB), immediately update or delete all tests that depend on the old source — `skipIf` guards on file existence mask the problem indefinitely. (Test Health Sprint)
- For server-rendered portals (MOHE, polycc, etc.), HTTP status codes are useless for link validation — always check rendered page content with a headless browser. (External Links Sprint)
- Never guess institution URLs from naming patterns — scrape the authoritative listing page instead. Even within a single system (e.g. IPG), URL conventions are inconsistent. (External Links Sprint)
- Design docs for rename/removal tasks must include a mechanical `grep` for all references before finalising the file list — manual estimates undercount because symbols propagate into tests, views, and utility code. (Subject Key Unification Sprint)
- Before changing a function signature, grep all call sites first — adding a new positional parameter breaks callers that pass options as the second argument. Use options objects for new optional parameters. (Saved Courses Sprint 1)
- When removing `as any`, check all types it was masking — a single `as any` often papers over multiple distinct type mismatches. Run `tsc --noEmit` after removal to catch cascading issues. (Quick Wins Sprint)
- When implementing a tech debt item, read the "What consistent looks like" field first — it defines the acceptance criteria. Skipping it leads to incomplete fixes that need a second pass. (Frontend Cleanup Sprint)
- When adding content for a trilingual project, always use the i18n system — never add content to DB fields that only support 2 languages. Check how many languages the project supports before choosing the storage method. (Tech Debt Quick Wins 2)
- Before writing the first frontend test in a project, verify that Jest is configured for the language (TypeScript needs `ts-jest` transform + path aliases in `jest.config.js`). An installed `jest` package is not enough — the config must exist. (IC Gate Sprint)
- When reusing a classifier or mapper across data sources, verify the semantics of each input column — same column name does not mean same data granularity. STPM `field` == aggregate `category`, unlike SPM where `field` is a specific sub-discipline. (Field Taxonomy Sprint 2)
- When building tools that process pipeline output, always run against real data before marking complete — synthetic test data can have different shapes (e.g. list of strings vs list of dicts) that only surface with production data. (Pipeline Sprint 3)
- Before creating migrations on a feature branch, always check `max(migration number)` on main — parallel branches creating same-numbered migrations is a recurring issue (third time in this project). Rebase or check before numbering. (Admin Auth Sprint)
- **The migration-number rule applies to TD numbers too** — a draft `TD-152` collided with another branch's `TD-152` (bursary donor) and had to be renumbered to `TD-153` mid-sprint. Before adding a TD, `grep 'TD-[0-9]' docs/technical-debt.md | sort` (or fetch+check main); numbering off a stale tree double-books. (Assignment-write-permission Sprint, 2026-07-02)
- **`next build`'s type-check OOMs on the 8 GB dev box after a full pytest run** — it prints "✓ Compiled successfully" then dies with a Jest-`WorkerError` in the parallel type-check phase. That's memory, not a code error. Confirm types with `npx tsc --noEmit --target es2018 --downlevelIteration --skipLibCheck` (bare `tsc` without the target flags surfaces bogus `TS2802 Set-iteration` errors in test files that Next's injected target resolves) and trust Cloud Build (more RAM) — it builds the same tree main does. Don't run the heavy pytest suite and `next build` back-to-back locally. (QC-gate Sprint, 2026-07-02)
- When accessing Stitch designs, use MCP `get_screen` instead of preview URLs — Stitch previews are not publicly accessible and will 404 in Playwright. (Identity Verification Sprint)
- Mark completed sub-tasks explicitly before moving on in long conversations — context loss causes duplicate work and user frustration when previously fixed code is "investigated" again. (Identity Verification Sprint)
- Never apply a fix pattern from one formula to another without independently verifying the target formula is actually wrong — "do the same for X" does not mean the same fix applies. Read the documentation block first. (Post-Launch Hardening Sprint)
- Display logic (how many items to show in a section) belongs in the frontend, not the backend — backend should rank and return a complete sorted list, not pre-split for a specific UI layout. (Post-Launch Hardening Sprint)
- For long-running subagent tasks (scrapers, deploys), verify `git log` before and after — concurrent subagents on the same branch can commit unexpected work while the main task is waiting. Use worktrees for isolation. (Pipeline Completion Sprint)
- When a user reports an "intermittent" bug, compare the failing path to a working path before hypothesising infrastructure causes — the differentiator reveals the actual bug. (i18n & Bug Fixes Sprint)
- Enforce security invariants (like "must have NRIC") via middleware, not per-endpoint checks — a single enforcement point prevents leaks when new endpoints are added and avoids the "check every time we implement a new feature" problem. (NRIC Hard Gate Sprint)
- After subagent work on frontend TSX files, always run `next build` before marking complete — TypeScript/JSX errors (like block-scoped variable ordering) are only caught by the compiler, not by reading the code. (NRIC Hard Gate Sprint)
- When a Django model field stores enum-like strings ("Ya"/"Tidak"), prefer BooleanField from the start — string booleans create type mismatches at serializer boundaries that are hard to diagnose. (i18n & Bug Fixes Sprint)
- localStorage is a cache, not a source of truth — always overwrite from the authoritative backend on login. Conditional writes (`if !localStorage`) cause stale-cache bugs that are hard to reproduce because they only affect returning users. (i18n & Bug Fixes Sprint)
- When fixing a data-type bug at a system boundary, trace the full pipeline (DB → API → frontend → cache → consumer → API) before starting — fixing one segment reveals the next, leading to multiple deploys for what should be one fix. (i18n & Bug Fixes Sprint)
- When applying Django migrations via raw SQL to bypass `InconsistentMigrationHistory`, verify the actual column types match the model AFTER applying — inserting into `django_migrations` only records the migration, it does not execute the DDL. A VARCHAR column read by a BooleanField silently coerces non-empty strings to `True`. (i18n & Bug Fixes Sprint)
- When adding a useEffect to a component with early returns (`if (x) return null`), place ALL hooks before ALL early returns — React's Rules of Hooks require hooks to be called in the same order on every render. A useEffect after an early return will crash when the condition changes. Spec reviewers should verify hook placement relative to conditional returns. (Auth Flow Canonical Refactor Sprint)
- Before drawing a data-driven conclusion that changes architecture (e.g. "use WhatsApp, not email"), verify the **authoritative** source — how the live system actually captures the field — not a convenience/back-office export. A blank or "N/A" column in an admin spreadsheet is not evidence of absence. (B40 Sprint 1)
- The frontend Jest config runs in **node** env (no jsdom, no React Testing Library). Put testable logic in pure modules (`src/lib/*.ts`) and unit-test those; don't write tests that render components — they have no DOM to render into. (B40 Sprint 2)
- When a sprint changes behaviour in an earlier sprint's code path (e.g. intake now also runs shortlisting), update that earlier sprint's tests in the same sprint, and run the **full** suite — a new-tests-only run hides the drift in the older tests. (B40 Sprint 3)
- Access an optional reverse OneToOne (e.g. `application.funding_need`) with `try/except Model.DoesNotExist` — Django's `RelatedObjectDoesNotExist` is **not** an `AttributeError`, so `getattr(obj, 'rel', None)` does not catch it and the access raises. (B40 Sprint 4a)
- The MANDATORY "prototype UI in Stitch before coding templates" rule applies to a multi-sprint build too, not just one-off pages — building the entire applicant + admin frontend across sprints without a single Stitch screen is the same miss at larger scale. Generate and get visual approval on the key screens before the first TSX file, even when the sprint plan reads as "backend + frontend together." (B40 Phase 1)
- When a field has two homes (e.g. financial data on both the application and the profile), name the single source of truth explicitly in the model docstring and make every other place read it live or snapshot it immutably — never store a second mutable copy. Two mutable copies of one fact have no defined winner and silently diverge. (B40 Phase 1.5a)
- Don't conclude a comms channel is dead from a back-office export — verify how the live system captures the field. ~600 students had verified Gmail (Google Sign-In) even though an admin spreadsheet's Email column was blank. (B40 Sprint 1)
