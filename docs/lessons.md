# Cross-Cutting Lessons

Lessons that apply to any future sprint, regardless of feature area.

- When testing one subsystem in a multi-rule scoring engine, provide explicit inputs for ALL subsystems — don't assume "empty" means "neutral". Default values in tag lookups (e.g. `c_tags.get('career_structure', 'volatile')`) can trigger unrelated scoring rules. (Sprint 4)
- When reusing legacy functions (e.g. Streamlit-era engine code) from a new context (e.g. Django serializer), verify the key/field naming conventions match. The serializer may produce different keys than the original caller expected. (Sprint 6)
- When `pd.concat` merges DataFrames with different columns, missing values become NaN (which is truthy in Python, not None). Always guard JSON-parsing functions with `isinstance(val, str)` before calling `.strip()` or `json.loads()`. (Sprint 7)
- Empty `subjects: []` in `subject_group_req` JSON doesn't mean "skip this rule" — it means "count from ANY subject". Test your engine logic with the actual data format, not just existing data patterns. (Sprint 7)
- When mocking a lazily-imported module (imported inside a function, not at module top), patch the actual module path (e.g. `google.generativeai.GenerativeModel`) not the local reference (e.g. `mymodule.genai`). The local name doesn't exist as a module attribute until the function runs. (Sprint 11)
- When filtering Django querysets by an external auth ID (e.g. Supabase UUID), use the FK traversal (`student__supabase_user_id=request.user_id`) not the FK column (`student_id=request.user_id`). The FK column is the integer PK, not the UUID — they'll never match. Always write view-level tests for auth-filtered endpoints, not just unit tests for underlying functions. (Sprint 12)
- Django's `.filter().update()` returns 0 silently when no rows match — it doesn't raise an error. If your loader needs to create-or-update (e.g. details for a course-institution pair that doesn't exist yet), use `update_or_create` instead. Silent no-ops in data loaders can orphan entire categories of data without any visible error. (Sprint 14)
- When your data pipeline has multiple loader steps that depend on each other (e.g. `links.csv` creates base records, `details.csv` enriches them), verify that ALL categories of data have coverage in the earlier step. If one category (e.g. TVET) is only in the later file, the pipeline silently skips it. (Sprint 14)
- When running multiple background agents that produce output for the same file, have them write to separate intermediate files (e.g. JSON) and merge in one pass — never let agents edit the same file concurrently. String-matching edits may not conflict, but file-level writes will. (Description Sprint)
- In TypeScript with a tsconfig target below ES2015, you cannot spread a `Set` directly (e.g. `new Set([...prev, id])`). Use explicit `Set.add()` instead: `const n = new Set(prev); n.add(id); return n`. The `--downlevelIteration` flag enables it but adds bundle size — prefer the explicit pattern. (Sprint 16)
- Supabase anon/service keys use HS256 (legacy JWT secret), but user access tokens from Supabase Auth use ES256 (JWKS). If your middleware only accepts HS256, all browser-initiated auth will silently fail. Always check the token's `alg` header and support both algorithms. Token length is a quick heuristic: ~300 chars = HS256, ~1300 chars = ES256. (Post-Sprint 17)
- When deploying to Cloud Run, verify ALL env vars are present — not just database/Django vars. Auth secrets (`SUPABASE_JWT_SECRET`), API keys (`GEMINI_API_KEY`), and service URLs (`SUPABASE_URL`) are equally critical. Maintain a checklist in CLAUDE.md and verify after every deploy. (Post-Sprint 17)
- Never deploy Cloud Run using `--image gcr.io/...` — gcr.io (old Container Registry) may contain stale images from failed builds. Always deploy from source (`--source .`), which uses Artifact Registry (`asia-southeast1-docker.pkg.dev`). If a build fails partway, the last-pushed image to gcr.io could be incomplete or from a different revision. (Post-Sprint 20)
- In Next.js 14, any page component that calls `useSearchParams()` must be wrapped in a `<Suspense>` boundary. Without it, the build fails during prerendering. The pattern is: export a wrapper component with `<Suspense fallback={...}>`, and put the real logic (with the hook) in an inner component. This applies to any client hook that depends on dynamic request data. (Post-Sprint 20)
- When the same value is computed in both frontend and backend, don't maintain two formulas — compute once and pass it through. Duplicate calculations inevitably drift (different inputs, different grouping logic) and produce subtle mismatches that are hard to spot in production. (v1.25.1)
- Don't sprint-close mid-session if there are more features planned for the same sprint. Sprint close should be the final action, not a checkpoint — doing it early creates confusing version numbers and duplicate changelog entries. (v1.32.0)
- Stitch MCP generates full pages from component-level prompts — if you need focused component comparison (e.g. "show me 4 variations of a progress bar indicator"), create a standalone HTML mockup instead. Stitch is better for full-screen designs, not isolated component exploration. (v1.25.0)
- When a dashboard has multiple rendering paths (e.g. ranked results with quiz vs flat list without), test both paths when adding new entries. Synthetic/injected items that work in one path can silently disappear in the other. (v1.32.2)
